This page describes how we intend the Sliced demo application to be used. It is part of the broader RealWorld Angular learning initiative.

About this app

Sliced is a sample pizza marketplace front end built with Angular. It exists to demonstrate patterns you might use in a real product—routing, forms, state, accessibility, and integration with a backend API—within a cohesive, opinionated codebase.

It is meant for education, experimentation, and as a reference implementation—not as a commercial service.

Personal information

Real names, addresses, phone numbers, and other identifying details are not required to try the app. We encourage you to use mock or fictional data (for example, made-up display names, delivery addresses, and contact fields) so you can learn the flows without exposing yourself or others.

If you choose to enter real personal data anyway, assume it could appear in logs, screenshots, backups, or shared environments—treat this site as a public sandbox.

Accounts, orders, menus, and other data may be reset, modified, or removed at any time. Do not rely on this environment for anything important.

For a dedicated setup, you can run the whole project—front end and backend—locally on your machine, or deploy your own instance by forking the RealWorld Angular repository and following its documentation.

Frequently asked questions

Is Sliced a real marketplace?
No. Sliced is a playground application for developers learning Angular and full-stack patterns. It is not a real marketplace: you cannot buy actual pizzas, no restaurant is obligated to fulfill orders shown in the UI, and there are no real-world consumer rights or merchant obligations attached to anything you do here.
Can I try checkout and orders like a real customer?
You need to create your own account to use signed-in features. You may then explore cart and checkout to learn how they work, but treat all actions as fictional. Any “payment” or order confirmation is part of the demo only and does not create a binding contract or real transaction.
Do I need to use my real name, address, or contact details?
No. Personal information is optional for exploring the demo, and mock data is encouraged so you do not risk sharing sensitive details in a learning environment.
What is the original RealWorld project, and how does RealWorld Angular fit in?
RealWorld is an open-source initiative (“the mother of all demo apps”) that defines a common API and encourages many reference front ends and back ends so developers can compare stacks fairly. RealWorld Angular is a fork of that idea that focuses on Angular only—a dedicated Angular learning and reference implementation rather than the multi-framework catalog of the upstream project. It is maintained by a former maintainer of the original RealWorld project.
Who is responsible for this software?
The application is provided as-is for learning. If you fork or deploy your own instance, you are responsible for your deployment, data, and compliance with applicable laws in your context.
Where can I learn more about the RealWorld initiative?
See the RealWorld Angular repository for documentation, issues, and how to run the stack locally.
Does this app use real payment processing?
No. The checkout flow collects delivery and billing addresses and order notes only. There is no credit card form, no payment gateway integration, and no real transaction of any kind. All orders are purely fictional—no money changes hands.
Why does the app ask for my address during checkout?
The address fields exist to demonstrate Angular’s signals-based reactive forms, form validation, and integration with an external geocoding service. When you type an address, your input is sent to Photon (a free, open-source geocoder by komoot) for autocomplete suggestions. No real delivery takes place.
Does this app show content that other users have typed?
No. Text input from one user is never displayed to other users. There are no reviews, comments, ratings, or any form of user-generated content visible to others. This is a deliberate design choice to avoid moderation risks, including exposure to NSFW content from bad actors. Images are also curated—users select from pre-uploaded pizza and pizzeria images rather than uploading their own files.
Why isn’t every feature of a real marketplace implemented?
Sliced is first and foremost an Angular learning playground. Features are added only when they provide meaningful opportunities to demonstrate Angular capabilities—signals, resource APIs, server-sent events, signals-based forms, lazy-loading, guards, and so on. The project deliberately avoids adding features just for the sake of completeness. If a feature does not showcase an interesting Angular pattern, it likely will not be built.
Does this project use any paid third-party libraries?
No. Every dependency is open-source and available under permissive licenses (MIT, Apache 2.0, and similar). There is no Stripe, no Sentry, no paid SDK of any kind. If your company would like its product showcased in a dedicated playground feature, sponsorship opportunities are available—contact the maintainer through the repository.
What version of Angular does this project use?
Sliced tracks the latest stable release of Angular (currently v21.2). It is an early adopter of new APIs—signals, httpResource, rxResource, and the signals-based reactive forms API—making it a living reference for modern Angular patterns and a useful migration guide for teams upgrading their own codebases.
